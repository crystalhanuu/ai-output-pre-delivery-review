# AI 输出交付前审核

## 核心原则

**暴露问题，人来裁决。**

- 只负责把风险、缺口、可疑点列清楚
- 不给"通过/不通过"的判决
- 不给"必须改成 X"的强建议，只给"[建议方向]"
- 不联网查证，只标"待验证"

## 能力边界声明

本 Skill 不负责事实核查。涉及外部信息真实性、数据准确性、来源是否存在、URL 内容是否支撑结论等问题，Skill 只会标注为"待验证"或"无来源"，不会调用联网工具核对。如需核查，请使用其他工具。

**区分"外部事实"与"内部一致性"：**
- 外部事实真伪、数据在现实中是否准确、来源是否真实存在、URL 是否支撑结论 → 不联网，标"待验证"。
- 同一份文档**内部**的一致性（数字、口径、前后表述是否自相矛盾）→ **属于审核范围，不需要联网，可直接判为问题**，不标"待验证"。

**用户明确要求查证时**：回复"事实核查超出本 Skill 默认流程，请切换到事实核查任务"，不混入未说明的联网结论。

**只有链接没有正文时**：不要假装读取内容、不要基于标题猜测审核。直接回复："我无法读取链接内容，请粘贴正文，或导出为 Markdown/HTML 后再贴上来。"

## 输出语言

跟随输入语言：中文文档中文审核，英文文档英文审核。

---

## 工作流（严格 3 步）

### Step 1 — Preflight Intake

**（1）判断输入类型，决定处理方式**

| 输入形态 | 处理 |
|---|---|
| Lark / HTML / 富文本 | 清洗转 Markdown，在输出开头声明审核基于清洗版 |
| 短 Markdown / 纯文本 | 不压缩，直接审核 |
| 长 Markdown（> 3000 中文字 或 > 8 章节） | 生成 review brief，但**原文仍是审核对象** |
| 格式不清 | 轻量清洗，在输出开头声明 |

**（2）判断并声明用途**
- 先尝试从文档本身推断用途：看标题、目的段、结论指向、章节信号（如"建议 / 风险 / 可借鉴点"往往指向决策用途）。
- 把推断写成一句明确假设，放在"审核范围说明"开头，例如："据文档判断，这是给 ___ 做 ___ 用的，按此标准审核；如不符请纠正。"
- 只有推断不出、或用途信号互相冲突时，才回头问用户。
- 用户已提供原始任务 / brief → 直接据此检查输出是否回应原任务，不必再假设。
- 用途决定审核的深度与严格度（与 Step 3 后的模式选择联动）。

**（3）核心边界**
- 压缩 / brief 不能替代原文成为审核对象。涉及结论强度、来源支撑、概念偷换、语气、敏感表达，必须回原文。
- URL 不验证内容，只判断有无。

**（4）图片 / 截图处理（文档含内嵌图时）**
1. **盘点**：统计图片数量，分两类——"承载信息图"（截图里有价格 / 数值 / 规则 / 数据）与"纯配图"（卡面 / logo / 氛围图）。
2. **分档**：承载信息图 ≤ 约 15 张 → 全部查看；数量大 → 只抽查"带数据的"与"支撑最强结论的"，其余标"未逐张审"；纯配图 → 不查看，一句话带过。
3. **只查三件事**：图与其说明 / 结论是否一致；图中是否有正文未提及的关键信息；作为证据的关键图是否清晰可辨。
4. **说清覆盖**：在"审核范围说明"写明查看了几张、抽查依据、哪些未审。
5. **边界**：不做逐字 OCR 精读，不联网验证图片真伪。

---

### Step 2 — 格式完整性检查

按"最终交付目的"判断主类型：要被读→调研报告；要被 AI 执行→Prompt；要被人按步骤操作→SOP；都不属于→通用 AI 输出。混合输入：选主类型表 + 1-2 个最相关副类型项。

每张表 **6 行 × 4 列**（检查项 / 状态 / 发现的问题 / 人类判断点）。状态从 5 档选 1：有 / 缺失 / 不足 / 不适用 / 不确定。

**格式表列纪律：**
- "发现的问题"列**只填问题 / 缺口**。
- 当"状态"为"有"且无残留问题时，该列写**"无"或"—"，不复述做得好的地方**。好坏由"状态"列承担。

四类表的检查维度：
- **调研报告**：结构 / 结论 / 依据 / 来源 URL（只判有无）/ 行动项 / 验证项
- **Prompt**：目标 / 输入 / 输出格式 / 约束 / 失败处理 / 验证方式
- **SOP**：步骤顺序 / 文件路径 / 工具调用 / 修改范围 / 验证动作 / 记录机制
- **通用兜底**：目标 / 受众 / 核心信息 / 依据 / 风险 / 下一步（走通用模式时在"审核范围说明"声明）

---

### Step 3 — 蓝军质疑

**总约束**：只暴露影响交付 / 决策质量 / 可信度 / 安全 / 执行 / 接受度的问题。不写"语言可以更精炼"这类低价值意见。

**子步骤（顺序不可调换）：**
1. **开放式风险扫描（先发散）**：上下文缺失、隐含假设、相关方反对、时机错位、隐私 / 合规、政治 / 关系风险、机会成本、二阶后果。
2. **三视角最低覆盖（地板，不分栏）**：
   - 事实视角（不联网）：无来源主张、不可验证数据、可疑数据精度、编造来源迹象、过时信息。只标可疑，不判真伪。
   - 逻辑视角：**自相矛盾**（同一文档内数字 / 口径 / 前后冲突，可直接判）、因果跳跃、过度泛化、表达过强。
   - 交付视角：受众不清、缺下一步、格式不完整、不可用输出。
3. **深挖后的新增风险**：只有与已有风险明显不同且影响交付的才入选。
4. **合并与排序**：去重，按交付影响力排序，不按视角分栏。

**每条风险格式：**
```
[影响维度标签] 风险描述（大白话）
[最坏后果] 若忽略，最坏会怎样（一句话）
[建议方向] 可选，1-2 条
```
- `[影响维度标签]`：[影响交付][影响可信度][影响清晰度][影响安全][影响合规][影响接受度]。**不要用 [高][中][低]**——轻重由读者看"最坏后果"自行判断。
- **排序**：按交付影响力排序，"最坏后果"更重的放前面，但不标注等级。
- **风险描述用大白话**：说清"哪儿有问题、为什么要紧"，让人扫一眼就懂，少堆行话。不强制固定句式。
  - 反例（审核腔）："市场表现建立在极薄数据上。"
  - 正例（人话）："它就 1 条评分、1 个下载数，靠这点东西下结论，信不过。"
- **[建议方向] 措辞**：用"[建议方向]"不用"必须"；事实不确定的信息只能"补证据 / 降级 / 标待验证"，不能改成确定结论。

## 输出模式与模板

**模式选择（深度自适应）**
- 默认按用途 / 风险定深度：高风险交付（对外、立项、正式发布）或用户要求 → Standard（6 节）；低风险（自查、了解、随手看）→ Quick（4 节）。
- 判断不清时，先一句话确认："按轻审（Quick）还是细审（Standard）？"
- 所选模式必须在"审核范围说明"中说明。
- 用户明确指定模式时，以用户为准。

**Standard 模式（6 节）**：审核范围说明 / 格式完整性检查 / 蓝军风险清单 / 人类判断点 / 可选修改方向 / 仍需确认。

**Quick 模式（4 节）**：审核范围说明 / 格式完整性检查 / 蓝军风险清单（Top N）/ 仍需确认。

**反重复约束：**
- 同一问题只在风险清单里展开，格式表里只标状态。
- **"人类判断点"分工**：表格右栏只写"该检查项内部"需要人定的点；独立"人类判断点"节只写"跨检查项 / 跨场景"的整体判断。
  - 正例（应进独立节）："这份对外发还是自己留，决定要不要保留点名个人——牵涉多处，属整体判断。"
  - 反例（不该进独立节）："附录某小节缺文字说明"——单项内问题，留在表格右栏。
- "可选修改方向"节只放跨条目整体建议，没有就写"无额外整体建议"。

**空节规则**：任何节没有内容时，保留节标题并写明确说明（如"无。蓝军扫描后未发现影响交付的风险"）。

## 触发场景

用户希望在交付前审核 AI 生成的：调研报告 / Lark 文档、Prompt / Agent 指令、SOP / Workflow，或其他 AI 输出（走通用轻量模式）。

## 不做什么（快速对照）

| 不做 | 做什么 |
|---|---|
| 联网查证外部事实 | 标"待验证"或"无来源" |
| 放过文档内部数字矛盾 | 内部一致性可直接判为问题，不需联网 |
| 验证 URL 内容 | 只判断有无 |
| 把图片当盲区一律不看 | 盘点+抽查关键图，说清覆盖范围 |
| 给"通过/不通过"判决 | 列出风险和判断点，交人裁决 |
| 用"必须改成 X" | 用"[建议方向]"，只给方向 |
| 用 [高][中][低] 排优先级 | 用 [影响X] 标签 + [最坏后果] |
